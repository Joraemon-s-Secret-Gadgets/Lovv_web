# 세부 일정 수정 채팅 기록 누적 설계

## 1. 배경

일정 생성 후 세부 일정 화면의 수정 챗봇에서 수정 요청을 연속으로 제출하면 일정 변경 API와 화면 반영은 정상 동작한다. 그러나 직접 입력한 수정 요청과 처리 결과가 대화 목록에 추가되지 않아, 다음 수정 시 이전 대화가 보이지 않는다.

현재 `PlanDetailView`는 플래너의 최근 메시지 최대 4개와 컴포넌트 로컬 메시지를 함께 렌더링한다. 로컬 메시지는 일정 카드의 빠른 장소 변경 동작에서만 추가되며, 수정 챗봇 입력 폼의 `handleFloatingChatSubmit` 경로에서는 추가되지 않는다. 처리 결과는 교체되는 단일 상태 문구인 `floatingChatNotice`로만 표시된다.

## 2. 목표

- 세부 일정 수정 챗봇에서 유효한 사용자 수정 요청을 제출 순서대로 누적한다.
- 각 요청의 처리 결과를 assistant 메시지로 누적한다.
- 새 요청이나 결과가 추가되어도 이전 수정 대화를 유지한다.
- 일정 변경, 후보 확인, 사용자 확인 필요, 입력 거절, API 실패 상태를 기존 동작과 일치하게 표현한다.
- 현재 세부 일정 화면 세션 동안 챗봇을 닫았다 다시 열어도 기록을 유지한다.
- 회귀 테스트로 두 번 이상의 연속 수정 대화가 모두 남는 것을 검증한다.

## 3. 비목표

- 수정 대화를 백엔드나 브라우저 저장소에 영구 저장하지 않는다.
- 새로고침 또는 다른 일정으로 이동한 뒤 대화를 복원하지 않는다.
- 추천 API 요청·응답 계약이나 백엔드 구현을 변경하지 않는다.
- 전체 플래너 채팅 기록과 세부 일정 수정 채팅을 하나의 서버 대화 모델로 통합하지 않는다.
- 채팅 UI의 시각적 재설계나 관련 없는 플래너 리팩터링을 수행하지 않는다.

## 4. 사용자 동작 및 요구사항

### R1. 사용자 요청 누적

- 사용자가 비어 있지 않은 수정 요청을 제출하면 해당 내용을 user 메시지로 한 번 추가한다.
- 같은 세부 일정 화면에서 다음 요청을 제출해도 이전 user 메시지를 제거하거나 덮어쓰지 않는다.
- 일정 카드의 빠른 장소 변경으로 생성된 기존 user 메시지 누적 동작도 유지한다.

### R2. 처리 결과 누적

- API가 수정 후보 또는 전체 수정안을 반환하면 결과를 설명하는 assistant 메시지를 추가한다.
- 일차 전체 변경처럼 사용자의 추가 확인이 필요한 요청은 확인 필요 결과를 assistant 메시지로 추가한다.
- 지원하지 않는 특정 장소 지정, 해석할 수 없는 명령, API 실패도 assistant 메시지로 추가하여 해당 요청의 결과를 기록한다.
- `floatingChatNotice`를 계속 사용하는 경우에도 대화 기록과 모순되는 문구를 표시하지 않는다.

### R3. 순서 및 중복 방지

- 각 대화 턴은 user 메시지 다음에 해당 assistant 결과가 오는 순서를 보장한다.
- 요청 하나당 user 메시지와 최종 처리 결과를 각각 한 번만 추가한다.
- 비동기 응답이 완료될 때까지 기존 기록은 유지한다.
- 기존 플래너 최근 메시지와 로컬 수정 메시지가 같은 ID를 사용하지 않도록 한다.

### R4. 세션 범위

- 수정 챗봇을 닫고 다시 열어도 `PlanDetailView`가 유지되는 동안 기록을 보존한다.
- 화면이 언마운트되거나 다른 일정 상세로 전환되면 로컬 기록이 초기화될 수 있다.

## 5. 설계

### 5.1 상태 소유권

`PlanDetailView`의 `localChatMessages`를 세부 일정 수정 대화의 단일 로컬 기록으로 유지한다. 진행 중인 대화를 서버에 저장하지 않는 현재 프론트엔드 정책과 일치하며, 수정 채팅 전용 상태를 상위 `usePlanner`로 끌어올리는 불필요한 범위 확장을 피한다.

### 5.2 메시지 추가 경로

작은 메시지 추가 helper를 두어 기존 배열을 함수형 상태 업데이트로 확장한다. 메시지 ID는 기존 `createLocalChatMessageId`를 사용해 각 추가 시 고유하게 만든다.

`handleFloatingChatSubmit`은 입력값이 비어 있지 않으면 user 메시지를 먼저 추가한다. 이후 검증·명령 해석·API 호출 결과에 따라 assistant 메시지를 추가한다.

기존 요청 함수는 UI 상태 문구와 후보 상태를 관리하고 있으므로, 이번 변경에서는 API 계약이나 후보 적용 흐름을 바꾸지 않는다. 요청 함수가 호출자에게 성공 여부 또는 필요한 결과를 반환하는 기존 구조를 우선 활용하고, 결과 문구는 현재 사용자에게 표시되는 문구와 동일한 의미를 사용한다.

### 5.3 데이터 흐름

1. 사용자가 수정 명령을 제출한다.
2. 입력값을 정규화하고 빈 입력을 차단한다.
3. user 메시지를 `localChatMessages` 끝에 추가한다.
4. 명령을 검증하고 수정 범위를 해석한다.
5. 필요하면 기존 `onRequestPlanModification` 경로로 API를 호출한다.
6. 후보, 반영 완료, 확인 필요, 입력 거절 또는 실패 결과를 assistant 메시지로 추가한다.
7. 기존 후보 확인 UI와 일정 변경 UI를 그대로 갱신한다.
8. 렌더링 시 플래너의 최근 메시지와 누적된 로컬 수정 메시지를 시간 순서 그룹으로 표시한다.

### 5.4 오류 처리

- 빈 입력은 메시지와 API 요청을 모두 생성하지 않는다.
- 지원하지 않는 명령은 user 요청 뒤에 assistant 안내를 남기고 입력값은 기존 UX와 일치하게 처리한다.
- API 예외는 기존 `getPlanModificationFailureMessage` 결과를 assistant 메시지로 남긴다.
- 실패하더라도 이전 대화와 일정 상태는 유지한다.
- 동일 결과가 상태 알림과 말풍선으로 동시에 보일 수 있으나, 대화 기록을 제거하거나 덮어쓰지는 않는다.

- `frontend/src/features/planner/PlanDetailView.test.tsx`
  - 연속 수정 요청과 결과 누적 회귀 테스트

## 7. 테스트 전략

### 자동 테스트

- 첫 번째 유효한 수정 요청 후 user 요청과 assistant 결과가 모두 보인다.
- 두 번째 유효한 수정 요청 후 첫 번째와 두 번째 요청·결과가 모두 남는다.
- 수정 챗봇을 닫았다 다시 열어도 누적 기록이 유지된다.
- 해석 불가 또는 지원하지 않는 요청은 user 메시지와 assistant 안내를 남긴다.
- API 실패 시 user 요청과 실패 assistant 메시지가 남고 이전 기록이 유지된다.
- 기존 장소 후보 확인 및 적용 동작이 계속 통과한다.

### 검증 명령

구현 계획에서 실제 `package.json` 스크립트를 기준으로 명령을 확정한다. 최소 검증 범위는 다음과 같다.

- `PlanDetailView` 관련 Vitest 테스트
- 프론트엔드 전체 단위 테스트
- TypeScript 빌드
- 린트
- 가능하면 V2 일정 Playwright 시나리오

## 8. 보안·개인정보·접근성

- 대화 내용을 새 저장소나 로그로 전송하지 않으므로 개인정보 보존 범위는 늘어나지 않는다.
- 실제 환경 변수나 토큰을 코드·테스트 데이터·커밋에 포함하지 않는다.
- 기존 user/assistant 정렬과 `aria-label="최근 대화"` 구조를 유지한다.
- 상태 안내를 말풍선으로 누적하더라도 기존 접근 가능한 상태 문구를 제거하지 않는다.

## 9. 수용 기준

- 한 일정 상세 화면에서 수정 요청을 두 번 이상 수행하면 모든 user 요청이 제출 순서대로 보인다.
- 각 요청 뒤에 성공, 후보, 확인 필요 또는 실패에 해당하는 assistant 결과가 보인다.
- 다음 수정이 이전 수정 대화를 덮어쓰지 않는다.
- 일정 변경과 후보 적용 기능은 기존처럼 동작한다.
- 새 회귀 테스트가 수정 전 실패하고 수정 후 통과한다.
- 관련 단위 테스트, 빌드 및 린트가 통과하거나 실행 불가 사유가 명확히 보고된다.
